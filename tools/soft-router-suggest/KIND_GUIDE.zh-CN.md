# Soft Router 六分类说明（前五类重点）

> 适用版本：当前 6 类简化版 Soft Router
> 
> 分类集合：`strategy / coding / vision / support / general / chat`
> 
> 本文重点说明前五类：
> - `strategy`
> - `coding`
> - `vision`
> - `support`
> - `general`

---

## 1. 文档目的

这份文档既是：

- **技术说明文档**：方便维护者理解五大类别的设计意图、命中逻辑和边界
- **用户指导文档**：方便使用者理解“什么问题更适合落到哪一类”

它不只是解释“分类名字”，更重点回答这几个问题：

1. 这个类别主要适合什么场景？
2. 它和其他类别的边界在哪里？
3. 为什么系统会把一段文本判到这一类？
4. 用户在配关键词时，应该往哪类加词？
5. 当前默认模型优先级为什么这样配置？

---

## 2. 六分类整体设计思路

当前 Soft Router 已从历史上的更细分类，简化为 6 类：

1. `strategy`
2. `coding`
3. `vision`
4. `support`
5. `general`
6. `chat`

这个简化方案的核心目标不是“语义最细”，而是：

- **更稳**：减少近义类别互相抢命中
- **更易维护**：关键词、模型优先级、用户 overrides 更容易管理
- **更符合真实使用**：大多数请求并不需要过细的路由颗粒度
- **更适合规则路由**：规则系统更擅长处理中等粒度类别，而不是太细碎的意图层级

其中：

- `strategy`：高层规划、方案、架构、决策、设计
- `coding`：代码、报错、调试、实现、工程问题
- `vision`：图片、截图、视觉识别、多模态判断
- `support`：日常帮助、办事建议、操作指导、轻量问题处理
- `general`：一般知识、解释、写作、翻译、通用任务
- `chat`：闲聊、陪伴、轻社交、无明确任务目标

---

## 3. 分类判定机制概览

当前每个 kind 的判定，主要依赖以下几类信号：

- **strong keywords**：强特征核心词
- **weak keywords**：辅助相关词
- **negative keywords**：排除词 / 其他类别强特征词
- **metadata signals**：如 `hasImage`、`hasCodeBlock`
- **thresholds**：如 `minScore`、`highScore`、`minStrongHits`
- **priority**：当多个类别分数接近时，优先级更高的类别更容易胜出

### 3.1 strong / weak / negative 的意义

#### strong
用于放置：
- 最能代表这个类别的强信号词
- 命中后就很可能属于该类的问题

#### weak
用于放置：
- 和该类别相关，但不够强的辅助词
- 单独出现时不一定能判定，但叠加多个后会增强该类得分

#### negative
用于放置：
- 排除词
- 或者其他类别的强特征核心词
- 防止某类“过于宽泛”而误吸收不属于自己的请求

# 4. 五大重点类别详解

---

## 4.1 `strategy` —— 策略规划类

### 4.1.1 适合场景

`strategy` 主要适合：

- 高层方案设计
- 路线规划
- 架构取舍
- 复杂任务拆解
- 项目推进策略
- 决策框架设计
- 中长期规划
- 系统级、产品级、业务级思考

### 4.1.2 典型问题类型

例如：

- 帮我设计一个三个月的产品推进路线图
- 这个系统是否应该拆成微服务？
- 做内部知识库权限系统，最小可行版本先做哪些层？
- 给我比较两个技术方案的优缺点
- 帮我梳理项目里程碑、风险点和资源安排
- 这个需求是先做 MVP 还是先补底层能力？

### 4.1.3 关键词特征

这类通常会带有：

- 方案
- 架构
- 规划
- 设计
- 路线图
- MVP
- 里程碑
- 权衡
- 取舍
- 风险
- 优先级
- 决策
- 策略

### 4.1.4 和其他类别的边界

#### 与 `coding` 的边界
- 如果重点是“具体代码怎么写 / 报错怎么修 / 实现细节怎么落地”，更偏 `coding`
- 如果重点是“应该怎么设计、怎么拆分、先后顺序怎么定”，更偏 `strategy`

#### 与 `support` 的边界
- `support` 更偏轻量帮助、办事建议、日常指导
- `strategy` 更偏复杂判断、结构化思考、方案级问题

#### 与 `general` 的边界
- `general` 可以回答一般解释类问题
- `strategy` 更强调面向目标、带决策意味、需要规划/权衡

### 4.1.5 当前默认模型倾向

当前关键词库中，`strategy` 默认模型优先级偏高如：

- `local-proxy/gpt-5.4`
- `local-proxy/gpt-5.2`

这意味着系统认为该类任务更适合：

- 更强的综合理解能力
- 更好的结构化组织能力
- 更好的方案比较和决策表达能力

### 4.1.6 适合新增到 strategy 的词

如果用户发现某些规划类文本没命中 `strategy`，通常适合往这里补：

- 强规划词
- 强方案词
- 强决策词
- 明显带项目级/架构级/路线级意味的词

不要把大量普通“解释”“帮助”“怎么做”都加到 `strategy`，否则会把 `support` 和 `general` 吸过来。

---

## 4.2 `coding` —— 编程开发类

### 4.2.1 适合场景

`coding` 主要适合：

- 代码报错排查
- 调试与修复
- 工程实现
- 构建/依赖/环境问题
- CI/CD 问题
- Git / 脚本 / 命令行 / 自动化问题
- 接口、模块、函数、项目开发问题

### 4.2.2 典型问题类型

例如：

- 这段 Python 报 `TypeError`，帮我定位原因
- `pip install -r requirements.txt` 为什么依赖冲突
- `node -v` 能用，但 VS Code 找不到 node，怎么排查
- 帮我实现一个上传图片后自动 OCR 的接口
- `git rebase` 和 `git merge` 有什么区别
- CI 一直失败，日志里提示 `ModuleNotFoundError`

### 4.2.3 关键词特征

`coding` 是当前规则最明显的一类之一。

强特征包括：

- `debug`
- `traceback`
- `stack trace`
- `TypeError`
- `ReferenceError`
- `ModuleNotFoundError`
- `build failed`
- `lint`
- `tsc`
- `npm`
- `pnpm`
- `pip`
- `.ts` / `.js` / `.py`
- `bug`
- `error`
- `报错`
- `代码`
- `code`

并且它还带有元数据增强：

- `hasCodeBlock = true` 时额外加分

### 4.2.4 和其他类别的边界

#### 与 `strategy` 的边界
- “怎么设计开发路线 / 架构怎么拆” → `strategy`
- “报错 / 实现 / 接口 / 构建 / 依赖 / 代码怎么改” → `coding`

#### 与 `vision` 的边界
- 如果问题中心是“看图识别”“截图诊断”“图里这个按钮/区域是什么”，更偏 `vision`
- 如果截图只是辅助，但问题本质是代码或系统故障，也可能仍归 `coding`

#### 与 `support` 的边界
- `support` 处理轻量帮助
- `coding` 面向工程技术处理，技术特征明显更强

### 4.2.5 当前默认模型倾向

假设当前 `coding` 默认模型列表偏开发向：

- `local-proxy/gpt-5.3-codex`
- `local-proxy/gpt-5.4`
- `local-proxy/gpt-5.2`
- `local-proxy/gpt-5.2-codex`

这类配置说明：

- 代码类任务优先尝试更适合编程/工程推理的模型
- 同时保留通用强模型兜底

### 4.2.6 适合新增到 coding 的词

适合补充：

- 报错名
- 异常名
- 开发工具名
- 构建链路词
- 语言/框架/文件扩展名
- 调试行为词

不建议把大量业务规划词、纯知识解释词放到 `coding`，否则会破坏边界。

---

## 4.3 `vision` —— 图像识别类

### 4.3.1 适合场景

`vision` 主要适合：

- 图片内容识别
- 截图诊断
- OCR 场景
- UI 截图辅助判断
- 图像中的按钮、区域、物体、文本识别
- 需要“看图”才能回答的问题

### 4.3.2 典型问题类型

例如：

- 帮我看看这张截图里哪个按钮是下一步
- 这张图片里的文字帮我识别一下
- 这块电路板发黑的位置像不像烧坏了
- 看下这张报错截图里最关键的问题点是什么
- 识别这张票据里的金额和日期

### 4.3.3 关键词特征与元数据特征

这类不仅依赖关键词，更强依赖元数据：

- `hasImage = true` 时，会获得很高加分

这意味着：

- 只要文本里没明确强烈指向其他类别
- 且消息确实带图
- `vision` 会非常容易成为命中类别

关键词常见包括：

- 图片
- 图像
- image
- photo
- 截图
- screenshot
- OCR
- 识别
- detection
- object detection

### 4.3.4 和其他类别的边界

#### 与 `coding` 的边界
- “有截图”不等于一定是 `vision`
- 如果核心问题是代码实现、报错修复，截图只是附属材料，可能还是 `coding`
- 但如果核心问题是“看图判断”，则应归 `vision`

#### 与 `support` 的边界
- 如果图像只是生活场景的一部分，比如“帮我看看这个界面下一步点哪里”，可能 `vision` + `support` 都有相关性
- 当前系统中，只要视觉信号强，通常优先归 `vision`

### 4.3.5 当前默认模型倾向

假设当前 `vision` 默认模型列表：

- `local-proxy/vision-model`
- `local-proxy/gpt-5.2`
- `local-proxy/gpt-5.4`

这体现了明显的多模态优先策略：

- 优先专门视觉模型
- 通用模型作为补充和兜底

### 4.3.6 适合新增到 vision 的词

适合补充：

- 图片相关实体词
- 截图操作词
- OCR / 检测 / 标注 / 识别类词
- 常见视觉任务名

不建议把纯技术错误词全堆到 `vision`，否则会抢走 `coding` 的请求。

---

## 4.4 `support` —— 日常支持类

### 4.4.1 适合场景

`support` 主要适合：

- 日常事务建议
- 轻量行动指导
- 办事流程建议
- 现实生活中的“下一步怎么做”
- 简单排查、简单操作建议
- 紧急但不一定技术复杂的问题处理

这是一个“**实用帮助型类别**”。

### 4.4.2 典型问题类型

例如：

- 我明天要去医院挂号、顺便办社保卡，帮我列个省时间顺序
- 显示器突然黑一下又恢复，优先怀疑什么
- 我今天事情很多，先做哪几件比较合理
- 这个流程我下一步先干嘛
- 某个系统/设备出了小问题，先检查哪些项

### 4.4.3 为什么 support 很重要

很多真实用户问题并不属于：

- 高层战略规划
- 编程开发
- 图像识别
- 纯知识问答

它们更像：

> “帮我把现实问题处理顺一点。”

所以 `support` 的存在，是为了接住：

- 轻任务
- 办事问题
- 流程建议
- 生活/工作中的小决策

### 4.4.4 和其他类别的边界

#### 与 `strategy` 的边界
- `strategy` 偏复杂、结构化、带规划深度
- `support` 偏轻量、实用、可立即执行

#### 与 `general` 的边界
- `general` 偏通用知识与普通任务
- `support` 偏现实帮助与操作建议

#### 与 `chat` 的边界
- `chat` 可以是闲聊、情绪陪伴
- `support` 则是带明确帮助目标的轻任务

### 4.4.5 当前默认模型倾向

假设当前 `support` 默认模型列表：

- `local-proxy/gpt-5.2`
- `local-proxy/gpt-5.4`
- `local-proxy/qwen3-coder-plus`

这类任务通常不需要最重型的复杂推理模型优先出场，但仍要保证：

- 表达清晰
- 行动建议实用
- 输出稳定

### 4.4.6 negative 的作用

`support` 中的 negative 很关键，它会压制一些明显更像：

- 深入分析
- 原理解释
- 方案设计
- 权衡比较
- 架构讨论

的请求，避免 `support` 把本该给 `strategy` / `general` 的问题抢过去。

### 4.4.7 适合新增到 support 的词

适合补充：

- 日常办事词
- 操作建议词
- 流程执行词
- 现实场景帮助词
- 轻量决策词

---

## 4.5 `general` —— 通用任务类

### 4.5.1 适合场景

`general` 是一个非常重要的“通用兜底+常规任务”类别。

它主要适合：

- 一般知识解释
- 翻译
- 改写
- 总结
- 概念说明
- 普通信息整理
- 不明显偏战略 / 编码 / 视觉 / 日常支持 的任务

### 4.5.2 典型问题类型

例如：

- `git rebase` 和 `git merge` 本质区别是什么
- 把这句话翻译成自然一点的英文
- 解释一下某个概念是什么意思
- 帮我润色一段说明
- 总结一下这段文字
- 给我一个简洁版定义

### 4.5.3 为什么 general 不能过弱

如果 `general` 太弱，会出现两个问题：

1. 大量普通问题被迫误判到 `support`
2. 一些知识解释类问题被误吸到 `strategy`

所以 `general` 的职责是：

- 承接“没有特别强类别特征，但又明显是任务型请求”的内容
- 作为稳定的通用任务层

### 4.5.4 和其他类别的边界

#### 与 `support` 的边界
- `support` 更偏现实帮助、流程建议、实操指导
- `general` 更偏知识、语言、说明、总结、翻译等通用任务

#### 与 `strategy` 的边界
- `strategy` 需要更强规划/权衡/方案意味
- `general` 则通常不涉及复杂决策

#### 与 `chat` 的边界
- `chat` 偏闲聊、陪伴、轻社交
- `general` 仍然是任务导向的

### 4.5.5 当前默认模型倾向

假设当前 `general` 默认模型列表主要是：

- `local-proxy/gpt-5.2`
- `local-proxy/gpt-5.4`

这类任务需要：

- 良好的通用表达能力
- 稳定的说明、整理、转述能力
- 不必像 `strategy` 那样优先重型规划能力，也不必像 `coding` 那样偏代码专长

### 4.5.6 适合新增到 general 的词

适合补充：

- 翻译类词
- 解释类词
- 总结类词
- 改写类词
- 通用文书和信息整理词

但注意不要把“办事建议词”“规划决策词”“代码异常词”混进来太多。

# 5. 五类之间的快速对照

## 5.1 一句话理解

- `strategy`：复杂规划与方案决策
- `coding`：代码与工程技术处理
- `vision`：看图、识图、截图判断
- `support`：日常帮助与轻量行动指导
- `general`：通用知识、翻译、总结、解释

---

## 5.2 快速决策表

### 当用户在问……

#### “应该怎么规划 / 怎么设计 / 怎么取舍”
优先看：`strategy`

#### “为什么报错 / 代码怎么改 / 环境怎么修”
优先看：`coding`

#### “帮我看这张图 / 这张截图 / 这张照片”
优先看：`vision`

#### “这件现实事务该怎么安排 / 下一步先做什么”
优先看：`support`

#### “帮我解释 / 翻译 / 总结 / 改写”
优先看：`general`

# 6. 用户如何判断自己该补哪一类关键词

如果用户发现某类文本经常命不中，可以这样判断：

## 6.1 先看问题中心是什么

不是看表面词，而是看**任务中心**：

- 中心是规划和方案 → `strategy`
- 中心是代码和工程处理 → `coding`
- 中心是视觉输入 → `vision`
- 中心是现实帮助和轻任务指导 → `support`
- 中心是通用说明和信息处理 → `general`

## 6.2 再决定加到 strong 还是 weak

### 加到 strong 的情况
- 这个词一出现，基本就很像这类

### 加到 weak 的情况
- 这个词和该类有关，但单独看不够强

### 加到 negative 的情况
- 这个词容易让该类误吸收别人的请求
- 或者它其实更像另一个类别的核心词

# 7. 技术维护建议

## 7.1 不要让某一类过度膨胀

尤其要注意：

- `support` 容易过宽
- `general` 容易变成“大杂烩”
- `coding` 容易因为报错词过多而吸走非代码技术问题

所以维护时应坚持：

- strong 少而准
- weak 多但克制
- negative 要能有效隔离边界

## 7.2 优先从误判样本反推关键词

维护关键词时，最好的方法不是凭感觉堆词，而是：

1. 收集真实误判文本
2. 看它为什么被错误命中
3. 决定是：
   - 给目标类补 strong/weak
   - 还是给错误类补 negative

## 7.3 模型配置要和类别职责一致

推荐原则：

- `strategy`：优先更强的结构化推理模型
- `coding`：优先代码/工程友好的模型
- `vision`：优先视觉模型
- `support`：优先稳定、轻快、实用的模型
- `general`：优先通用表达均衡的模型

# 9. 最终总结

当前前五类的职责可以概括为：

- `strategy`：负责“想清楚怎么做”
- `coding`：负责“把技术问题搞定”
- `vision`：负责“看图再判断”
- `support`：负责“把现实事情处理顺”
- `general`：负责“把一般任务说明白、整理好、表达清楚”

如果这五类边界保持清晰，那么整个 Soft Router 会具备几个明显优点：

- 路由更稳
- 误判更少
- 模型切换更合理
- 用户更容易理解和维护
- 后续做关键词微调时更可控
